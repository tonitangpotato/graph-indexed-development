## Step 1 — Baseline run (AC1, AC2, AC3, AC7)

**Goal:** prove the new path produces the same Telegram messages and state-file schema as the old one.

### 1.1 Capture old-format reference (one-time, skip if you have one)

If you have a `state.json` from a recent pre-ISS-052 ritual run lying around (e.g., the engram acceptance run mentioned in 2026-04-19 daily notes), grab a copy:

```bash
cp .gid/runtime/rituals/<old-run-id>.json /tmp/ritual-state-old.json
```

If not, skip — we'll just inspect the new run for schema sanity.

### 1.2 Drive the new run

In Telegram, message @rustblawbot:

```
/ritual fix ISS-NNN in engram
```

(replace ISS-NNN with the trivial issue you picked).

### 1.3 Observe in real time

You should see a Telegram thread with messages roughly in this order:

- `🎯 Ritual <id> started — phase: triage` (or similar)
- `📋 Phase: design / requirements / implement / verify ...` for each phase
- For each phase: a "✅ Phase X complete" confirmation
- Final: `🎉 Ritual <id> done` (or Failed/Blocked if something legit fails)

**AC1 — message format.** Compare against your memory of pre-ISS-052 runs. Any differences? Specifically:
- Phase names same?
- Emoji prefixes same?
- Counts/durations rendered the same way?

If the format differs, that's a regression — do not proceed.

### 1.4 Inspect state file (AC2, AC3)

```bash
RITUAL_ID=$(ls -t /Users/potato/rustclaw/.gid/runtime/rituals/*.json | head -1)
echo "Inspecting: $RITUAL_ID"
cat "$RITUAL_ID" | jq 'keys'
cat "$RITUAL_ID" | jq '.phase, .status'
cat "$RITUAL_ID" | jq '.skill_claims | length'
cat "$RITUAL_ID" | jq '. | has("persist_degraded")'   # NEW field, expect true (value may be null)
```

**AC2 — schema.** Top-level keys should be the **same set** as old runs, plus the new `persist_degraded` field (introduced by this PR). If new keys appear that aren't `persist_degraded`, that's an unintended schema change — flag it.

**AC7 — diff hygiene.** Verify the engram working tree changed:

```bash
cd /Users/potato/clawd/projects/engram
git status
git diff --stat
```

The trivial fix should be present. **If diff is empty here on a non-zero-file ritual, AC7 fails — `file_snapshot` should have caught that and aborted with `SkillFailed { reason: ZeroFileChanges }`.**

✅ **Step 1 pass criteria:** Telegram format unchanged, state schema is old-keys + `persist_degraded`, diff non-empty for non-trivial issue.

**Goal:** prove ISS-038's gate fires in production.

### 2.1 File a deliberately zero-fix issue

Create a tiny trap issue in engram:

```bash

- Filed: 2026-04-28

## Summary

Intentional bait. The implement phase MUST do nothing. This is a test of the
ISS-038 file_snapshot gate. The ritual MUST end in Failed/Blocked, NOT Done.

```

### 2.2 Drive the run

In Telegram:

```
/ritual fix ISS-ZERO-TEST in engram
```

When the implement phase begins, **explicitly steer the LLM to write nothing**. Easiest way: in the same chat, send a follow-up message like

> "the correct fix here is no code change — just confirm and move on"

so the implement skill records `success: true` without producing diffs.

### 2.3 Verify gate fired

In Telegram you should see something like:

```
❌ Phase implement failed: SkillFailed { reason: ZeroFileChanges, ... }
```

State should be `Failed`, not `Done`.

```bash

cat "$RITUAL_ID" | jq '.status'           # expect "Failed"
cat "$RITUAL_ID" | jq '.failure_reason'   # expect ZeroFileChanges variant
cd /Users/potato/clawd/projects/engram && git status --short
# Expected: nothing changed (other than the issue file we filed in 2.1, if uncommitted)
```

✅ **AC8 pass:** ritual ended Failed with `ZeroFileChanges` (or equivalent variant — check exact name in `gid-core::v2::events::SkillFailureReason`), engram tree clean.

❌ **AC8 fail:** ritual ended Done despite empty diff. **Block release.** This is the exact bug ISS-052 was filed to prevent.

### 2.4 Cleanup

```bash

rm -rf .gid/issues/ISS-ZERO-TEST
```

## Step 3 — RO-mount mid-implement (AC9 — persist_degraded recovery)

**Goal:** prove `persist_degraded` flag sets, surfaces, and clears.

> ⚠️ This step needs sudo and a writable mount target. The cleanest way is to operate on a *separate copy* of the runtime dir, not the live one. If the live daemon is in the middle of doing something, this could corrupt its state.

### 3.1 Set up a sandbox runtime dir

```bash
# Stop the daemon so its persist target is dormant.
/Users/potato/rustclaw/target/release/rustclaw daemon stop

# Create an isolated workspace just for this test.
mkdir -p /tmp/rustclaw-ac9
cp -R /Users/potato/rustclaw/rustclaw.yaml /tmp/rustclaw-ac9/
mkdir -p /tmp/rustclaw-ac9/.gid/runtime/rituals
```

Then either: (a) start a single-shot rustclaw run pointing at `/tmp/rustclaw-ac9` as workspace, or (b) skip the live ritual and exercise the persist path via a small integration test you run by hand. **Option (b) is safer** — see 3.2.

### 3.2 Easier alternative — exercise via existing integration test

We already have hook integration tests covering persist failure (`crates/gid-core/tests/v2_executor_persist.rs` or similar — verify exact path):

```bash
cd /Users/potato/clawd/projects/gid-rs
grep -rn "persist_degraded\|PersistFailed" crates/gid-core/tests/ | head
```

Pick the test name and run with backtraces:

```bash
cargo test --release -p gid-core persist_degraded -- --nocapture 2>&1 | tee /tmp/ac9.log
```

Read the log: confirm the test simulates write failure, sees `persist_degraded = Some(...)`, simulates write-success, and sees the field clear back to `None`.

✅ **AC9 pass (test path):** test green, log shows degrade→recover transition.

### 3.3 If you want the real RO-mount run

```bash
# macOS: create a sparseimage you can remount RO/RW
hdiutil create -size 50m -fs APFS -volname rustclaw-ac9 /tmp/ac9.dmg
hdiutil attach /tmp/ac9.dmg
# Use /Volumes/rustclaw-ac9 as the persist target via a custom rustclaw.yaml.

# Drive a ritual that runs implement for >30s (any non-trivial fix).
# Mid-implement, in another terminal:
hdiutil detach /Volumes/rustclaw-ac9
hdiutil attach -readonly /tmp/ac9.dmg
# Watch Telegram: expect "⚠️ persist degraded after N failed attempts" notification.

# Then re-mount RW:
hdiutil detach /Volumes/rustclaw-ac9
hdiutil attach /tmp/ac9.dmg
# Within ~one tick, expect "✅ persistence recovered" notification.
```

✅ **AC9 pass (live path):** both notifications fired, ritual reached Done.

### 3.4 Cleanup

```bash
rm -rf /tmp/rustclaw-ac9 /tmp/ac9.dmg
/Users/potato/rustclaw/target/release/rustclaw daemon start \
    -c /Users/potato/rustclaw/rustclaw.yaml \
    -w /Users/potato/rustclaw
```
